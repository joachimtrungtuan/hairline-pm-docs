# gen-code-map Skill + Semantic Code Maps (replacing tree dumps)

**Date**: 2026-06-26
**Type**: New skill (in `skills-engineering/`) + code-map redesign + script reorganization
**Scope**: `skills-engineering/gen-code-map/` (new), `project-notes/code-map-*.md`,
`project-automation/scripts/`, `local-docs/INDEX.md`
**Initiative**: token-optimization, Phase 2 follow-up

## Why

The Phase 2 code maps were generated with `repomix --no-files` — i.e. plain directory trees,
no better than `ls -R`, and they go stale on every commit. Replaced them with **curated,
folder-group-level semantic maps** that bind each folder group to project module codes
(P-/PR-/A-/S-) and FRs. Folder-group altitude is deliberate: folders change rarely, so the
maps age slowly, while individual files churn constantly.

## Changes

### 1. Semantic code maps (replace the tree dumps)
Rewrote the three persistent maps as curated folder-group maps — each group lists Purpose,
Modules, FRs, and Entry-point files. Generated by a sub-agent reading `routes/api.php`, the
FR→module table in `system-prd.md`, and the tech-spec architecture sections (verified against
route prefixes). Sizes: `code-map-backend.md` ~204 lines / 24 groups (Laravel API, serves all
3 tenants); `code-map-frontend.md` ~116 / 15 (web — **Provider + Admin**); `code-map-app.md`
~105 / 13 (**Flutter — Patient only**). `main/` was read-only throughout.
Findings recorded in the maps: web frontend is Provider+Admin (not patients); Flutter app is
Patient-only; the tech-spec "React Native" mobile note is **stale** (actual stack is Flutter).

### 2. New `gen-code-map` skill (`skills-engineering/gen-code-map/`)
Created with explicit user approval to add to the FROZEN skills folder. Codifies the
generate/refresh workflow as a user-triggered (BLOCKING once invoked) skill:
- `SKILL.md` — purpose, the 3 persistent map paths (codebase ↔ tenant table), hard rules
  (read-only `main/`, folder-group altitude, accuracy-over-coverage, never persist signatures),
  and the workflow: detect drift → refresh only changed groups → verify → accept snapshot → log.
- `references/generation-rules.md` — map output format, module/FR binding inputs, altitude.
- `scripts/check-code-map-drift.sh` — deterministic drift detector (folder diff vs snapshot;
  no LLM, no file reads). Modes: default check, `accept`. Snapshots seeded:
  backend 84 / frontend 221 / app 146 folders in `project-notes/.code-map-snapshots/`.

### 3. Script reorganization (cleanup)
- Deleted `project-automation/scripts/regen-code-maps.sh` (superseded — it produced the tree
  dumps we replaced).
- Moved drift detection into the skill (`gen-code-map/scripts/check-code-map-drift.sh`); the
  standalone copy in `project-automation/scripts/` was removed.
- `project-automation/scripts/` now holds only `search.sh` + `README.md`; README rewritten to
  drop the repomix/regen section and point code-map work to the `gen-code-map` skill.

### 4. INDEX.md
- §A.2 Skill Directory: added the `gen-code-map` row.
- §B Tools: code maps now "maintained by the `gen-code-map` skill", not a script.
- §D Code Map: rewritten — semantic folder-group maps (codebase↔tenant table), on-demand
  signature command, and drift/refresh via the skill. Removed the tree-only repomix commands.

## Design note

Drift detection stays a **deterministic script** (folder diff — zero tokens); map *refresh*
is the **skill** (judgment — reads code, binds modules/FRs). The skill calls the script. This
keeps the cheap part cheap and codifies the repeatable LLM workflow, matching how every other
operation in this project is run as a skill.

## Refinement (minor, same day)

- Closed a seam in the `main/` read guidance: the older "Reading `main/`" tiers in
  `CLAUDE.md`/`AGENTS.md` only pointed to `system-technical-spec.md` / `system-data-schema.md`
  and never named the code maps. Added an explicit instruction that reading `main/` —
  **including for code review / code analysis** — starts from the code maps (`INDEX.md` §D),
  then opens only the named files (signatures on demand). Reworded INDEX §A row to
  "Read / review / analyze main/ source code … → map first → open only the named files".

- Clarified the code maps' role across all touch points: a **locator, not evidence** — they
  pinpoint WHERE to look to prevent broad searches, but findings must be confirmed by reading
  the actual source. Added this statement to each map's header, the `gen-code-map` generation
  rules (so future maps carry it), INDEX §D, and the `CLAUDE.md`/`AGENTS.md` main/ read guidance.
